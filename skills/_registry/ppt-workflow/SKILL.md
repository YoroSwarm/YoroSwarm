# PPT 分阶段制作工作流（HTML 幻灯片版）

## 概述

将 PPT 创作视为**分阶段的团队工作**，而非一次性生成。核心流程：

**需求澄清 → 资料调研 → 大纲规划 → 策划稿 → HTML 幻灯片 → 复核**

关键原则：**约束工作流程，而非约束实现方式**。

## ⚠️ 输出格式：HTML 幻灯片（强制要求）

**所有演示文稿必须以单一 HTML 文件形式交付。**

### 强制规格

1. **单文件**：所有幻灯片页面包含在一个 `.html` 文件中，CSS/JS 全部内联
2. **16:9 比例**：每张幻灯片画布固定 1920×1080 像素，通过 CSS `transform: scale()` 自适应屏幕
3. **导航功能**：
   - 键盘：← → 方向键翻页，Space/Enter 下一页，Backspace 上一页，Home/End 首尾页，Escape 显示总览
   - 鼠标：点击屏幕左/右 20% 区域翻页，滚轮翻页
   - 触摸：左右滑动翻页
4. **页码指示器**：底部显示当前页/总页数和进度条
5. **页面过渡**：平滑的 CSS 过渡动画
6. **全屏支持**：按 F 键或双击可切换全屏模式

### ⚠️ 分步生成策略（必须遵循）

**禁止一次性生成完整 HTML 文件！** 由于 LLM 输出长度限制，必须按以下两步工作：

#### 第一步：生成框架文件（使用 `create_workspace_file`）
生成包含完整 HTML 结构的框架文件，其中每张幻灯片使用**占位符内容**：
- 完整的 `<head>`（CSS 样式、全局配色方案）
- 完整的 `<script>`（导航逻辑、缩放逻辑）
- 每张 slide 的 `<div class="slide">` 容器，内部放置**唯一可识别的占位标记**
- 占位标记格式：`<!-- SLIDE_CONTENT_N -->` （N 为页码数字）

示例框架中的一页：
```html
<div class="slide" data-slide="3">
  <!-- SLIDE_CONTENT_3 -->
</div>
```

#### 第二步：逐页填充内容（使用 `replace_in_file`）
对每张幻灯片，使用 `replace_in_file` 工具将占位标记替换为实际的 HTML 内容：
```
replace_in_file({
  path: "presentation.html",
  replacements: [{
    old_str: "<!-- SLIDE_CONTENT_3 -->",
    new_str: "<h1>实际标题</h1>\n<div class='content'>实际内容...</div>"
  }]
})
```

**每次调用只替换 1-3 页**，避免单次输出过长。

#### 为什么必须分步
- LLM 单次输出有 token 限制（通常 4096），完整 HTML 幻灯片远超此限制
- 分步生成保证每一步都在限制内完成
- `replace_in_file` 的精确替换确保不会破坏已有内容
- 出错时只需要重新替换单页，不需要重新生成整个文件

### 为什么用 HTML

- 无需任何外部依赖或软件即可查看（浏览器即可）
- 支持丰富的排版、动画、交互效果
- 天然支持响应式，适配不同屏幕
- 便于版本控制和协作修改
- 可直接在 Swarm 的文件预览中查看

### HTML 幻灯片模板

请参考 `references/prompts.md` 中的「HTML 幻灯片生成」模板，其中包含完整的 HTML 结构框架。**每次生成幻灯片时必须使用该模板作为基础结构。**

## 核心方法

1. **从问题出发，而非从模板出发** — 先搞清楚给谁看、为什么做、希望对方记住什么
2. **内容先行，设计随后** — PPT 的灵魂是内容，不是皮囊
3. **增加中间层：策划稿** — 在大纲和成品之间插入页面级规划卡片
4. **分层交付** — 根据需求在合适的层级停下（大纲/策划稿/样例/全套方案）

## 环境能力评估

开始工作前，评估当前环境能否：
- 获取或验证外部事实（搜索、网页抓取等）
- 处理用户提供的参考材料
- 创建结构化中间产出
- 生成 HTML 幻灯片文件
- 向用户交付文件

仅用于判断可达到的交付层级，不用于锁定具体工具选择。

## 输入模式

### 1. 仅主题
例："帮我做一个 AI 安全的 PPT"

需补充收集：受众、目的、页数范围、风格偏好、是否需要调研、是否需要中途审阅。

### 2. 主题 + 简报
用户已提供受众、目的、风格、页数和关键信息。

### 3. 素材驱动
用户提供了 URL、报告、PDF、笔记等参考资料。以此为主要上下文，仅在能显著提升质量时补充外部调研。

### 4. 已有大纲或草稿
用户已有结构或初稿，希望改进而非重做。

## 交付层级

按需在合适层级停下：

1. **调研底稿** — 事实、证据、开放问题、来源
2. **大纲** — 章节结构、页面标题、页面目标
3. **策划稿** — 逐页规划卡片（目标、信息层级、证据需求、布局方向）
4. **样例制品** — 2-3 页 HTML 幻灯片样例，用于确认方向和风格
5. **完整 HTML 幻灯片** — 包含所有页面的单一 HTML 文件，含导航和动画
6. **审阅意见** — 逻辑、证据、密度、一致性、重点问题

## 标准工作流程

### 第 1 步：澄清简报
收集避免方向性错误所需的最小输入：
- 主题、受众、目的
- 页数范围、风格/调性
- 必须包含的章节、必须避免的方向
- 用户希望快速粗稿还是分阶段确认

### 第 2 步：判断是否需要调研
以下情况需要调研：
- 主题依赖时事、市场数据、产品详情、统计数据或技术事实
- 用户明确要求研究支撑
- 用户提供的素材明显不完整

如不需要，直接从已有素材出发。

### 第 3 步：收集/整理上下文
利用可用能力收集或整理支撑材料。如无法获取外部信息，明确说明结果受素材范围限制。

可参考 `references/prompts.md` 中的「主题调研」提示词模板。

### 第 4 步：产出调研底稿
在构建大纲前，将工作知识归纳为简洁底稿：关键事实、支撑证据、受众相关上下文、风险与待确认事项。

### 第 5 步：生成大纲
仅在底稿和上下文足够好之后创建大纲。

可参考 `references/prompts.md` 中的「大纲架构师」提示词模板。

要求：
- 保持逻辑结构严密
- 论点与可用证据对齐
- 适合演讲表达，而非仅供阅读
- 便于用户审阅

### 第 6 步：生成策划稿
对于非简单主题，不要直接从大纲跳到成品页面。

逐页创建规划卡片，包含：
- 页面目标（观众应记住什么）
- 核心信息与信息层级
- 支撑证据/数据需求
- 推荐表达方式（对比/流程/时间线/数据卡/象限/卡片网格等）
- 布局方向与设计注意事项

可参考 `references/prompts.md` 中的「策划稿生成」提示词模板。

### 第 7 步：生成样例 HTML 幻灯片
在全面扩展前，使用分步策略生成 2-3 页样例幻灯片：

1. 用 `create_workspace_file` 创建包含 2-3 页占位符的框架 HTML
2. 用 `replace_in_file` 逐页填充封面页和 1-2 个内容页

让用户确认：
- 整体视觉风格是否满意
- 配色方案是否合适
- 排版布局是否清晰
- 动画过渡是否恰当

**必须使用 `references/prompts.md` 中的 HTML 幻灯片模板结构。**

### 第 8 步：重要演示需要审阅关卡
以下场景应设置审阅关卡：
- 对外展示的演示
- 管理层/董事会汇报
- 客户/销售演示
- 基于不确定事实的内容
- 复杂技术主题

### 第 9 步：扩展为完整 HTML 幻灯片
仅在方向获得确认后才全面扩展。**严格遵循分步生成策略：**

1. **生成完整框架**：用 `create_workspace_file`（或 `replace_workspace_file` 覆盖样例）创建包含所有页面占位符的框架
2. **逐批填充内容**：用 `replace_in_file` 每次填充 1-3 页内容，直到所有页面完成
3. **验证完整性**：用 `read_workspace_file` 读取文件，确认所有 `<!-- SLIDE_CONTENT_N -->` 占位符已被替换

确保：
- 所有页面均使用统一风格
- 导航功能完整（键盘/鼠标/触摸）
- 页码指示器和进度条正常工作
- 过渡动画流畅
- 16:9 比例在不同屏幕上正确显示

### 第 10 步：复核与披露限制
交付前检查：
- **内容**：逻辑、事实置信度、证据覆盖、信息密度、重点与层级、章节一致性
- **技术**：HTML 文件可独立打开、导航功能正常、16:9 比例正确、无外部依赖
- **视觉**：文字可读性、配色一致性、留白合理、无溢出或截断

如能力限制影响了结果，坦诚说明。

## 协调规则

- **调研规则**：任务依赖事实时，先收集/验证再自信发言
- **诚实规则**：环境不支持理想工作流时，说明限制，在可达的最高价值层级继续
- **审阅关卡规则**：高风险任务在全面扩展前先展示中间成果
- **HTML 优先规则**：最终产物必须是单一 HTML 文件，16:9 比例，含完整导航

## 质量标准

成功使用此 Skill 应产出：
- **单一 HTML 文件**，浏览器打开即可演示
- **16:9 比例**幻灯片，自适应屏幕大小
- **完整导航**：键盘、鼠标、触摸均可操作
- 论证清晰的演示结构
- 更少的装饰性空洞页面
- 调研存在时更好的事实支撑
- 通过策划稿获得更好的可控性
- 用户可在全面扩展前审阅的中间产物
- 更少的"AI PPT"套模板感
